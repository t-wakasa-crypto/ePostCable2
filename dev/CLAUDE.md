# 開発パイプライン運用ルール（dev/）

> 本ファイルはプロジェクトルートの `CLAUDE.md`（ルーター）から「開発進行管理に
> 関する作業」と判定された場合に読み込まれる。design/・dev/・src/ は同一
> プロジェクト内に共存するため、design/CLAUDE.md（設計パイプライン章）とは
> 独立した内容だが、コピー配置は不要（同じリポジトリ内の別ファイルとして直接参照する）。

## 概要

`dev/` パイプラインは、`design/` で確定した仕様をもとに、
`implementation-planner` → `implementer` → `code-reviewer` のループを
Claude Code が**ユーザーに逐一確認を取らず完全自律**で回し、Laravel 実装を
完成させる仕組みである。停止するのは、開発環境未整備・レビュー往復の上限到達・
設計との齟齬・全タスクがブロックされている、のいずれかに該当したときのみ。

**この開発パイプラインに関する作業を行う前に、必ず
[docs/システム開発パイプライン設計書.md](../docs/システム開発パイプライン設計書.md)
を Read し、そこに記載された手順・判定条件に従うこと。** セッション開始時の
状態確認手順、開発環境の確認方法、自律ループの継続・停止条件、機能追加の
自動反映、アドホック修正フロー、タスク進捗の更新ルール、レビュー後の修正
ループなど、実際の運用ルールはすべて同書に記載されている。本ファイルには
重複して書かない。

## 更新ルール

開発パイプラインの手順・エージェント構成・停止条件・自動反映の仕組みなどに
追加・変更を行った場合は、**必ず
[docs/システム開発パイプライン設計書.md](../docs/システム開発パイプライン設計書.md)
を同時に更新すること**。本ファイル（薄いポインタ）は、参照先や概要が変わる
場合を除き更新不要。

## 禁止事項
- `docs/システム開発パイプライン設計書.md` を読まずに開発パイプラインの作業を
  進めること
- 開発パイプラインの運用ルールを本ファイルに書き足し、
  `docs/システム開発パイプライン設計書.md` と内容が二重管理・不整合になること
- パイプラインを変更したのに `docs/システム開発パイプライン設計書.md` を
  更新しないまま放置すること
