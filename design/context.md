# 対話で収集した情報

## システム概要
顧客への**請求書**および**納品書**を PDF で生成し、メールで自動配信する管理システム（Laravel製）。

- **何を作るか**: 基幹システムからの出荷データ取得 → 納品書・請求書レコードの自動作成 → バッチ／キューによる PDF 生成・メール配信 → 管理画面での送信状況確認・手動再送・各種設定管理。
- **誰が使うか**: 主な利用者は**営業・カスタマー担当**（送信状況の確認・再送）。加えて**システム管理者**（admin 権限：バッチ手動起動・ユーザー管理・システム設定）。
- **なぜ作るか**: 請求書・納品書の作成〜PDF化〜メール送付までの**手作業を自動化**し、送付漏れ・誤送信を防ぎ、送信状況を追跡できるようにするため。

## 既知の制約・前提
- 技術スタックは確定済み（リバース対象）：
  - PHP ^8.3 / Laravel ^13.7
  - PDF: barryvdh/laravel-dompdf ^3.1（A4縦・日本語フォント）
  - フロント: Tailwind CSS（Vite）
  - DB: **SQL Server（sqlsrv）**（開発・本番とも。開発も外部 `db-sv03.solid-corp.local` に接続、DBコンテナは持たない）
  - キュー: Redis / キャッシュ・ロック: database（既定）または redis / セッション: database
  - メール: SMTP（開発は Mailpit）
  - 基幹連携: Laravel HTTP Client（`BACKBONE_API_URL`、未設定時はダミーデータ）
  - テスト: PestPHP ^4.7
  - 実行環境: Docker Compose（php / nginx / redis / mailpit / worker / scheduler）
- 通知先・BCC は `.env` ではなく `system_settings` テーブル（システム設定画面）で管理。
- 成果物の保存先は `design/` 配下のみ（プロジェクト規約）。

## 競合調査の要否
**不要**。既存の自社実装システムのリバース・検証が目的のため、競合・他社サービス調査は対象外。

## 優先度・スコープ
今回のゴールは2点（フェーズ＝**実装済み**）：

1. **要件 vs 実装の検証**: 実装が機能的に要件を満たしているかを確認したい。
   → 既存仕様・実装コードを照合し、要件カバレッジ／不足・乖離を洗い出す。
2. **仕様書のリバース**: 既存の詳細仕様書（`請求書メール配信システム仕様.md`）を、設計パイプラインの上流成果物（requirements / basic-design / detailed-design 等）へ再構成する。

### 既存の主要機能（リバース元のスコープ）
- 出荷データ取得バッチ（基幹システム連携、重複スキップ、消費税10%算出）
- 請求書／納品書メール送信バッチ（limit・stuck-timeout・retry-failed オプション、二重起動防止ロック）
- キュージョブによる PDF 生成・メール送信（system_settings 由来のリトライ／タイムアウト設定）
- 管理画面：ダッシュボード・請求書/納品書一覧&詳細・送信状況フィルタ・手動再送・一括再キュー・PDF/CSV ダウンロード
- メール送信履歴（親=バッチ単位 / 子=書類1通単位、ポリモーフィック関連）・出荷取得履歴
- 管理者通知メール（バッチ結果レポート）・全メールBCC
- ユーザー管理（admin/general、退職管理、自己削除防止）・システム設定（テストメール送信含む）
- 認証・認可（auth / admin ミドルウェア、退職者ログイン拒否、ログイン試行回数制限、CSRF）

### 除外・対象外
- 競合調査、新規システム設計
- 画面からの金額・明細編集機能（現在・将来ともに提供しない。2026-07-01のヒアリングで確定〈Q-02〉。請求データは基幹システムの売上確定値をそのまま使用し、変更があり得るのは入金予定日のみ）

## その他メモ
- リバース元: `請求書メール配信システム仕様.md`（作成 2026-06-11 / 最終更新 2026-06-25、対象: invoice / src）。詳細設計レベルまで網羅済み。
- ステータス体系: 書類は pending → processing → sent / failed / failed_permanent。出荷取得ログは running → completed / failed。
- 注意点（仕様書記載）: `tax`=税率(%)、`tax_amount`=税額(円) で混同しやすい。送信状態判定は `failed_at` 最優先。手動再送は当日分の `manual-resend` 親に集約。
- **検証方針＝仕様ベース**: 実装コード（src/）はこのプロジェクト配下に存在しないことを確認済み。よって code-reviewer による実コード照合は行わず、仕様書の記述から「要件カバレッジ・設計の自己整合性・記述の漏れ／矛盾」を検証する。矛盾・不明点は `design/questions.md` に記録する。
