# システム開発パイプライン設計書（dev/）

> 本書は [dev/CLAUDE.md](../dev/CLAUDE.md) から参照される詳細仕様書である。
> `dev/CLAUDE.md` は薄いポインタとして維持し、実際の手順・判定条件はすべて
> 本書に記載する。**開発パイプラインの手順・エージェント構成・停止条件などに
> 追加・変更を行った場合は、必ず本書を更新すること**（`dev/CLAUDE.md` 側は
> 参照先が変わらない限り更新不要）。

## この章の目的：Claude Code による完全自律開発

開発パイプラインは、`design/` で確定した仕様（requirements.md /
detailed-design.md / db-design.md）をもとに、**Claude Code がサブエージェントを
用いて完全自律的にシステムを開発する**ための仕組みである。人間が1行ずつコードを
書く前提ではなく、以下のループを Claude Code 自身が**ユーザーに逐一確認を取らずに**
回し続けることで実装を完了させる。

```
implementation-planner（実装計画の生成。未生成なら自動で最初に起動）
  → implementer（設計と実装計画に基づき src/ へ自律実装。テスト実行含む）
  → code-reviewer（設計適合性・品質・セキュリティのレビュー）
  → 総合評価が「要修正」「差し戻し」→ implementer を自動的に再起動して修正 → 再レビュー
  → 総合評価が「OK」→ 開発完了
```

メインチャット（このセッション）の役割は、上記ループを**ユーザーの追加指示を待たずに
連続して回すこと**であり、実装コードそのものを書く役割は `implementer` が担う。
「開発パイプラインを開始して」「実装して」と言われたら、以後は下記「自律ループの
継続条件・停止条件」に該当するまでエージェントの起動判断を待たない。ただし、これは
「毎回のエージェント起動を許可制にしない」という意味であり、後述の禁止事項・
停止条件（設計との齟齬・環境未整備・ループ上限）は例外なく適用する。

## 開発環境の確認（実装着手前の前提条件）

`implementer` による自律実装（マイグレーション実行・テスト実行を含む）は、
`src/` の実行環境が整っていることを前提とする。この環境構築は
`scripts/setup.sh`（Docker起動・Composerインストール・`.env`配置・DB初期化・
フォント設定・Nodeセットアップ・フロントビルドを一括実行）が担当する。
環境が未整備のまま `implementer` を起動しないこと。

### 確認方法

| 項目 | 確認コマンド／方法 | 未整備と判断する条件 |
|---|---|---|
| `.env` | `src/.env` の存在を Glob で確認 | 存在しない |
| Composer依存関係 | `src/vendor/autoload.php` の存在を Glob で確認 | 存在しない |
| Dockerコンテナ | Bash で `docker compose ps --format json` を実行し、`docker-compose.yml` に定義された全サービス（現時点: nginx / php / redis / mysql / worker / scheduler / mailpit。変更されうるため実際の定義を都度参照する）の State が `running` であることを確認 | 1つでも `running` でないサービスがある、またはコマンド自体がエラー終了する（Docker Desktop / dockerd が起動していない場合を含む） |
| DBマイグレーション | Bash で `docker compose exec -T php php artisan migrate:status` を実行 | コマンドが失敗する（DB未接続・未インストール）、または `Pending` のマイグレーションが残っている |
| フロントエンドビルド（必要な場合のみ） | `design/detailed-design.md` でフロントビルドが必要と判断される場合のみ、`src/public/build` の存在を確認 | 必要なのに存在しない |

### 分岐
- いずれか1つでも未整備 → 不足項目を具体的に列挙し、「開発環境が未整備です
  （不足: ◯◯）。`./scripts/setup.sh` を実行してから再度お知らせください」と
  報告する。この状態では `implementer` を起動しない（自律ループの停止条件）。
- すべて整っている → 確認なしで `implementer` を起動する（後述の
  「セッション開始時の動作」Step 2 に従う）。

### 確認のタイミング
- `implementer` を起動する直前に毎回行う（コンテナ停止など、環境の状態は
  セッションをまたいで変化しうるため、初回のみで済ませない）。
- 修正モードで `implementer` を再起動する場合も同様に確認する。

## 自律ループの継続条件・停止条件

完全自律で回し続けるループにも、無人暴走を防ぐための明確な停止条件を設ける。
以下のいずれかに該当した場合は、ループを止めてユーザーに報告し、次の指示を待つ
（このときだけは確認を取ってよい、というより取らなければならない）。

- **レビュー往復の上限到達**：`code-reviewer` の総合評価が「要修正」または
  「差し戻し」のまま **3回連続**で解消されない場合（＝`implementer` による
  修正後の再レビューが3回続けて OK にならない）。回数はこの開発パイプライン
  実行1回分（このセッション内の連続実行）でカウントし、報告時にリセットする。
- **開発環境が未整備**：「開発環境の確認」で1項目でも未整備と判定された場合。
  `scripts/setup.sh` の実行をユーザーに依頼し、実行完了の連絡を待つ。
- **設計との齟齬**：`implementer` が設計書と矛盾する実装が必要だと判断した場合
  （「タスク進捗の更新ルール」参照）。`design/questions.md` に記録した上で
  ユーザーに判断を仰ぐ。
- **依存関係でブロック**：`dev/implementation-plan.md` の未完了タスクが、
  他の未解決事項（外部システムとの調整待ち等）によりすべてブロックされていて
  `implementer` が着手できるタスクがない場合。

上記に該当しない限り（＝通常の「要修正」でのやり直しや、次のタスクへの着手）は、
ユーザーに起動可否を確認せず、メインチャットが自動的に次のエージェントを起動する。

（この上限「3回」は暫定値。運用してみて多い/少ないと感じたら本書のこの節を
調整する。）

## セッション開始時の動作

開発関連の作業を行う際は、**ユーザーに話しかける前に必ず開発の進捗状態を確認する**。
何かを尋ねる前に、以下の Step 1〜3 を省略しないこと（design/CLAUDE.md の
「セッション開始時の動作」と同じ考え方を、実装フェーズ向けに適用したもの）。

### Step 1: 状態確認
- `design/` 配下を Glob し、開発に最低限必要なファイル
  （`requirements.md` / `detailed-design.md` / `db-design.md`）が
  揃っているか確認する。
- `docs/システム開発環境設計書.md` を Read し、「定義すべき内容」が未整備の
  プレースホルダのままでないか確認する（判定基準は同書冒頭を参照）。
- `design/feature-log.md` が存在するか Glob で確認する。存在すれば、
  「開発への反映」が「未反映」のエントリ（＝機能追加が設計側では完了しているが
  実装計画にまだ取り込まれていない）がないか確認する（詳細は後述「機能追加の
  自動反映」）。
- `dev/adhoc-fixes.md` が存在するか Glob で確認する。存在すれば、未完了
  （`- [ ]`）の項目がないか確認する（詳細は後述「アドホック修正フロー」）。
- `dev/implementation-plan.md` を Read し、タスクのチェックボックス
  （`- [ ]` 未完了 / `- [x]` 完了）の件数と完了率を数える。
- `dev/review-report.md` が存在するか Glob で確認する。存在すれば
  ファイル冒頭の `**総合評価**: ...` の値（OK / 要修正 / 差し戻し）を確認する。

### Step 2: 状態に応じた分岐
「開発パイプラインを開始して」「実装して」等の起動指示を受けた場合、以下の分岐は
ユーザーに起動可否を確認せず自動的に実行する（「自律ループの継続条件・停止条件」
に該当する場合を除く）。単に進捗を尋ねられただけの場合は、分岐結果の報告のみ行い
起動はしない。

- **`design/` に必要なファイルが揃っていない（存在しない、または一部欠けている）**
  → 不足しているファイル名を明示し、「先に design/CLAUDE.md の設計パイプラインを
  完了させてください」と報告して停止する（自律ループの停止条件）。
- **`design/` に必要なファイルが揃っている が `dev/implementation-plan.md` が
  存在しない** → `implementation-planner` を自動的に起動し、実装計画を生成する。
  生成後、続けて次の分岐（未完了タスクがある）に進む。
- **`docs/システム開発環境設計書.md` が未整備のプレースホルダのまま**（実装計画の
  有無に関わらず、これが未整備だと `implementer` は必ず Step 1 で停止する）→
  `implementer` を起動せず、**メインチャットがユーザーとヒアリングして
  同書を埋める**（design/CLAUDE.md の初期ヒアリングと同じ考え方で、技術
  スタック・Lint設定・ディレクトリ構成・テスト方針・命名規則・コミット規約を
  対話で確認し、`docs/システム開発環境設計書.md` に反映する）。これを行う
  専用のサブエージェントは存在しないため、メインチャット自身の責務とする。
  整備が完了したら次の分岐（未完了タスクがある）に進む。
- **`design/feature-log.md` に「開発への反映: 未反映」のエントリがある**（初回の
  実装計画が既に存在する場合） → `implementation-planner` を追記モードで
  自動的に起動する（「機能追加の自動反映」参照）。追記後、続けて次の分岐
  （未完了タスクがある）に進む。
- **`dev/adhoc-fixes.md` に未完了の項目がある** → `implementer` をアドホック
  修正モードで自動的に起動する（「アドホック修正フロー」参照）。
- **未完了タスクがある**（チェックが付いていない項目が残っている） →
  先に「開発環境の確認」を行う。環境が整っていれば `implementer` を自動的に
  起動する。着手するタスク（未完了の先頭、または依存関係上着手可能なもの）は
  `implementer` 自身が判断する。環境が未整備なら起動せず、`scripts/setup.sh`
  の実行をユーザーに依頼して停止する（自律ループの停止条件）。
- **全タスク完了 かつ `dev/review-report.md` が存在しない** → `code-reviewer`
  を自動的に起動する。
- **`dev/review-report.md` の総合評価が「要修正」または「差し戻し」** →
  「開発環境の確認」を行った上で `implementer` を自動的に再起動する（後述の
  「レビュー後の修正ループ」に従う。上限到達時は自律ループの停止条件に従う）。
- **`dev/review-report.md` の総合評価が「OK」** → 開発完了である旨を報告して
  ループを終了する。

### Step 3: 状況報告
このセッション・ターンで最初にユーザーへ発言する際は、必ず次を含める。
1. 進捗（n/m タスク完了、レビュー状態）
2. 次に何をする必要があるか、またはこれから自動的に何を行うか

進捗確認をせずに「何をしますか？」と一般的に尋ねて終える動作は禁止する。
自律ループが継続している間（Step 2 で自動起動を続けている間）は、エージェント
1回完了するごとに状況報告のために止まる必要はない。「自律ループの継続条件・
停止条件」に該当したとき、またはループが完了（OK）したときに報告すればよい。

## 機能追加の自動反映（design 側の機能追加を Laravel 実装に繋げる）

`design/CLAUDE.md` の機能追加フロー（機能追加・画面追加・テーブル追加・
ボタン追加など、設計変更を伴うあらゆる変更が対象）は、`design/` 配下の
設計ファイルを更新するところまでで完結し、そのままでは実装計画・実装コードに
自動反映されない。この章は、その先を**自律ループの一部として**繋ぐための
ルールである。

- `design/feature-log.md` のエントリが「状態: 設計反映済み」「開発への反映:
  未反映」になった時点で、次に開発パイプラインが起動されたとき（ユーザーが
  何かを尋ねただけでも、Step 1 の状態確認で）これが検知される。
- 検知したら、Step 2 の分岐に従い `implementation-planner` を追記モードで
  自動的に起動する。ユーザーに「実装計画を更新しますか？」等の確認は取らない
  （既存の完了チェックを壊さない追記モードであるため、確認なしで進めてよい）。
- 追記モードの `implementation-planner` は、既存タスクの完了チェックを保持
  したまま新機能分のタスクだけを `dev/implementation-plan.md` に追加し、
  `design/feature-log.md` の「開発への反映」を「反映済み」に更新する。
- その後は通常の自律ループ（未完了タスクがある → `implementer` 起動 →
  `code-reviewer` → …）に合流し、新機能分のタスクも自動的に実装される。
- `implementer` は、feature-log.md 由来のタスクをすべて完了・チェックした
  時点で、対応する feature-log.md エントリの「開発への反映」を「実装完了」に
  更新する。

これにより、「design で機能追加 → 何もしなくても次に開発パイプラインが
動いたときに Laravel コードへ反映される」という流れが成立する。

## アドホック修正フロー（設計変更を伴わない直接修正）

バグ修正・軽微な文言修正・「動かないので直して」等、**設計内容の変更を
伴わない**直接的なコード修正は、機能追加フロー（design 側）を経由せず、
この章に従って直接 `dev/` 側だけで処理する。

### トリガー
- 「〇〇のバグを直して」「〇〇が動かないので直して」「〇〇を修正して」等、
  具体的な修正内容が伴う発言。
- 設計・要件の変更を伴うかどうか判断が難しい場合は、design/CLAUDE.md の
  「機能追加フローの対象外（アドホック修正）」に従い `AskUserQuestion` で
  確認してから、このフローか機能追加フローかを振り分ける。

### 手順
1. `dev/adhoc-fixes.md` が存在しなければ新規作成し、修正内容を
   `- [ ] AD-xx: 修正内容の要約` の形式で追記する（番号は全体で一意）。
   複数件をまとめて依頼された場合は複数行に分けて記録する。
2. `implementer` をアドホック修正モードで自動的に起動する（`implementer` は
   Step 0 で `dev/adhoc-fixes.md` の未完了項目を検知し、これを最優先の
   対象として修正する）。
3. 修正完了後、`dev/adhoc-fixes.md` の該当項目を `- [x]` に更新する
   （`implementer` の責務。「タスク進捗の更新ルール」と同じ考え方）。
4. `code-reviewer` を自動的に起動して検証する（通常のレビュー後の修正ループに
   合流する）。

### dev/adhoc-fixes.md の形式
```markdown
# アドホック修正ログ

- [ ] AD-01: 〇〇画面で△△するとエラーになる不具合を修正
- [x] AD-02: ボタンのラベル文言を「送信」から「登録」に修正
```

過去の項目は削除せず、修正履歴として積み上げていく（questions.md や
feature-log.md と同様、追記のみで運用する）。

## タスク進捗の更新ルール

タスクの実装そのもの、および実装完了に伴うチェックボックスの更新は
`implementer` エージェントの責務である（本ファイルはそのルールを定義し、
`implementer` はこのルールに従って自律的に更新する）。

- 各タスクの実装が完了したら、**その都度** `dev/implementation-plan.md` の
  該当するチェックボックスを `- [x]` に更新する。まとめて後で更新するのではなく、
  1タスク完了ごとに更新すること（中断時に進捗が失われないようにするため）。
- 実装中に計画外のタスクが必要になった場合は `dev/implementation-plan.md` に
  タスクを追記してからチェックする。計画に記録せず実装だけを進めることは禁止する。
- タスクの完了基準に迷う場合は、対応する `detailed-design.md` / `db-design.md` の
  記述に立ち返って判断する。設計と食い違う実装をした場合は、実装を設計に合わせるか、
  `design/questions.md` に記録するかをユーザーに確認する（design/CLAUDE.md の
  「論点解消フロー」に相当する対応が必要な場合があるため）。メインチャットは
  `implementer` からこの確認要求が報告された場合に、ユーザーへの確認を代行する。

## レビュー後の修正ループ

1. `code-reviewer` を起動し `dev/review-report.md` を生成する。
2. 総合評価が「要修正」または「差し戻し」の場合、確認を取らず `implementer` を
   自動的に再起動する（`implementer` は Step 0 で `dev/review-report.md` を
   検知し、自動的に修正推奨事項に従って実装を直す「修正モード」で動作する）。
   このとき往復回数を1つ数える。
3. 修正によって完了状態が変わったタスクがあれば `implementer` が
   `dev/implementation-plan.md` のチェック状態も見直す（再び未完了になる場合は
   チェックを外す）。
4. 修正が完了したら `code-reviewer` を自動的に再起動し、`dev/review-report.md`
   を最新版で上書きする（過去のレビュー履歴は残さない。経緯を残したい場合は
   ユーザーに確認してから別途メモする）。
5. 総合評価が「OK」になるまで確認なしで 1〜4 を繰り返す。ただし往復回数が
   3回に達しても「OK」にならない場合は、「自律ループの継続条件・停止条件」に
   従ってループを止め、`dev/review-report.md` の内容とここまでの経緯を要約して
   ユーザーに報告する。

## src/CLAUDE.md との関係
`implementer` エージェントは、`design/CLAUDE.md`・本書の進行管理ルールに
加えて `src/CLAUDE.md`（コーディング規約）を必ず併読して適用する。本書は
「何を・どの順で進めるか」、`src/CLAUDE.md` は「どう書くか」を扱う、役割の異なる
2つのファイルである。

## 禁止事項
- 進捗確認をせずにユーザーへ話しかけること
- タスク完了後にチェックボックスを更新しないまま次のタスクに進むこと
- 計画にないタスクを記録せず実装すること
- `dev/review-report.md` が「要修正」「差し戻し」のまま、ユーザーへの報告なしに
  「開発完了」として扱うこと（自律ループ中の再挑戦は可。完了扱いにするには
  総合評価が「OK」であることが必須）
- `src/CLAUDE.md` のコーディング規約を確認せずに実装を進めること
- `implementer` を使わず、開発パイプライン外でコードを直接編集して進捗を
  記録しないまま済ませること（自律開発ループの追跡が崩れるため）
- 「開発環境の確認」を省略して `implementer` を起動すること（環境未整備での
  マイグレーション・テスト失敗を実装バグと誤認するおそれがあるため）
- 「自律ループの継続条件・停止条件」に該当した状態（レビュー往復3回到達・
  環境未整備・設計との齟齬・タスクがすべてブロック）で、ユーザーに報告せず
  ループを継続すること
