> 本書は [src/CLAUDE.md](../src/CLAUDE.md) から参照される詳細仕様書である。
> `src/CLAUDE.md` は薄いポインタとして維持し、実際のコーディング規約はすべて
> 本書に記載する。**コーディング規約・ディレクトリ構成方針などに追加・変更を
> 行った場合は、必ず本書を更新すること**（`src/CLAUDE.md` 側は参照先や役割の
> 説明が変わらない限り更新不要）。

> `artisan`・`app/` 等はすべてこの配下に置く）。

## 現在の状態

このテンプレートのひな型規約は、実運用中の Laravel プロジェクト
「請求書メール配信システム」（`請求書メール配信システム仕様（ひな型用）.md`
参照）の技術スタック・実装パターンをもとに整備している。個々のプロジェクトで
実装を開始する際は、`design/detailed-design.md`・`design/db-design.md` の
内容に応じて本書の内容を見直し、プロジェクト固有の値に更新すること
（本書の内容をそのまま使ってよい項目と、プロジェクトごとに決め直すべき項目が
混在するため、下記の各節で区別して記載する）。

## 2. ディレクトリ構成方針（モジュール型アーキテクチャ）

**本テンプレートでは、小規模なシステムであってもモジュール型構成を採用する**
（開発者ごとに実装場所がばらつくことを防ぎ、複数人・複数プロジェクトで
開発手段を統一するため）。Laravel標準のフラットな `app/` 構成
（`app/Models/`・`app/Http/Controllers/` に全機能を並べる構成）は採用しない。

### 採用パッケージ

`nwidart/laravel-modules` を用いて `Modules/` 配下に機能単位でモジュールを
切る。プロジェクト固有の事情で別のモジュール化手段を使う場合は、その理由と
採用パッケージを本書に追記すること。

### モジュール分割の基準

- 業務ドメイン単位で分割する（例: 請求書配信システムなら
  `Modules/Invoice`・`Modules/DeliveryNote`・`Modules/ShipmentFetch`・
  `Modules/SendMailLog`・`Modules/SystemSetting`・`Modules/User`・
  `Modules/Auth` 等）。
- 複数モジュールから共通で使う横断的なコード（基底クラス・共通トレイト・
  共通例外等）は `Modules/` に置かず `app/` 直下（`app/Support/` や
  `app/Exceptions/` 等）に置く。
- 1モジュールが肥大化しすぎる場合（画面数・テーブル数の目安は
  `design/detailed-design.md`・`design/db-design.md` の規模による）、
  サブドメイン単位でさらに分割することを検討する。

### モジュール内部の標準構成

```
Modules/{ModuleName}/
├── app/
│   ├── Console/Commands/       # バッチ・Artisanコマンド
│   ├── Http/
│   │   ├── Controllers/
│   │   └── Middleware/         # モジュール固有のミドルウェアのみ
│   ├── Jobs/                   # キュージョブ
│   ├── Mail/                   # Mailable
│   ├── Models/
│   └── Services/                # ビジネスロジック（Controllerを薄く保つ）
├── database/
│   ├── migrations/
│   ├── factories/
│   └── seeders/
├── resources/views/
├── routes/
│   └── web.php
└── tests/
    ├── Feature/
    └── Unit/
```

コントローラは薄く保ち、業務ロジックは `Services/` に切り出す方針は
請求書メール配信システムの `PdfService`・`ShipmentFetchService` の実績パターンを
踏襲する。

## 3. 認証方式

本プロジェクト（請求書メール配信システム）は社内利用者のみを対象とし、
**外部利用者向け認証は存在しない**。認証方式は以下のとおり確定している
（`design/requirements.md` FR-16 / NFR-S-01〜04 参照）。

### 3.1 利用者（社内スタッフ）の認証

- **認証方式**: メールアドレス＋パスワード（Laravel 標準 Auth）。
  Entra ID SSO は採用しない。
- **パスワード**: ハッシュ保存・min:8・confirmed バリデーション。
  `users` テーブルの `password` カラムに保持する。
- **ロール**: `role` カラムで `general`（担当者）/ `admin`（管理者）を区別する
  （`auth` / `admin` の2段ミドルウェアで画面を保護）。
- **退職者**: `retired_at` がセット済みのユーザーは、認証情報が正しくてもログインを拒否する。
- **ユーザー管理**: `User` モデル（`users` テーブル）で一元管理。
  外部ユーザー用のモデル・ガード分離は不要。

### 3.2 セキュリティ方針

- ログイン試行回数制限（`RateLimiter`）: メールアドレス＋IP単位で失敗5回まで、
  超過時は約60秒ロック（成功でリセット）。
- 全 POST リクエストへの CSRF トークン検証。
- 未認証ユーザーは `/login` へリダイレクト。
- セッションは database ドライバに保存。

## 4. コーディングスタイル

- Lint/Formatter: Laravel Pint（標準プリセット）を使用する
  （`scripts/artisan-pint.sh` 参照）。
- ステータス管理: 状態を持つモデルは、文字列リテラルを直書きせず
  `STATUS_PENDING` 等のクラス定数として定義する（請求書配信システムの
  `Invoice::STATUS_*` パターンを踏襲）。
- スコープ: よく使う絞り込み条件はEloquentのローカルスコープ
  （`scopePending()` 等）として定義する。
- 例外: 「リトライ不可能な失敗」を表す独自例外（`RuntimeException` 継承）を
  ジョブ内で使い分ける（`PermanentJobFailureException` の実績パターン）。
- サービスクラス: 複雑なビジネスロジック・外部API連携は `Services/` に切り出し、
  コントローラ・ジョブから呼び出す。

## 5. テストの書き方・配置ルール

- テストフレームワークは **PestPHP** を使用する。
- 各テストは `RefreshDatabase` トレイトでDBをリセットし、
  `database/factories/`（モジュール内）のファクトリでテストデータを生成する。
  ファクトリには `running()`・`failed()` 等の意味のある state を定義する。
- テストファイルは対応するモジュールの `tests/Feature/`・`tests/Unit/` に配置し、
  プロジェクト直下に一括配置しない。
- 各テストファイル・各テストケースには、初心者にも分かる日本語コメント
  （ファイル冒頭のdocブロックと各テストの説明）を付与する。
- カバレッジ方針: `design/requirements.md` の各 FR/NFR の受入条件に対応する
  テストを必須とする（`code-reviewer` エージェントのレビュー観点と一致させる）。

## 6. 命名規則

- 変数・メソッド名: キャメルケース（`camelCase`）。
- クラス名: パスカルケース（`PascalCase`）。
- ステータス等の定数: スネークケース大文字（`STATUS_FAILED_PERMANENT` 等）。
- マイグレーション・DBカラム名: スネークケース（`sent_at` 等）。DB設計の
  命名規則は `.claude/skills/db-design-standard` の基準に従う。
- モジュール名: パスカルケース単数形（`Invoice`・`DeliveryNote` 等）、
  対応する Eloquent モデル名と一致させる。

## 7. コミット単位・コミットメッセージの規約

- 1コミット1論理変更を基本とする（複数機能の変更を1コミットに混在させない）。
- コミットメッセージの先頭に変更種別を付ける（例: `feat:`・`fix:`・`refactor:`・
  `test:`・`docs:`）。
- `scripts/git-commit.sh` の対話的ステージ機能を使い、意図しないファイルの
  混入を防ぐ。

## 8. 設計とコードの対応関係

- `design/detailed-design.md` のAPI定義は、対応するモジュールの
  `routes/web.php` およびコントローラのdocコメントに、エンドポイントの
  参照元（章番号等）をコメントで残す。
- `design/db-design.md` のスキーマ定義とマイグレーションのカラム定義・型・

  設計・実装のどちらを正とするかをユーザーに確認する。

## 9. データベース接続（開発/本番でドライバを切り替え可能にする方針・複数DB接続）

### 9.1 背景・方針

社内の既存システムは SQL Server で稼働しているが、開発時の手軽さを優先し
Docker 環境には **MySQL** をコンテナとして同梱する。本番では MySQL・SQL Server
のどちらでも稼働できるようにし、環境変数だけでドライバを切り替えられる状態を
維持する。

- **アプリ自身のメインDB**（このシステムが新規に持つテーブル群）は、
  Laravel の Eloquent／Query Builder を用いた**ドライバ非依存な書き方**を徹底し、
  `.env` の `DB_CONNECTION` を `mysql` ⇄ `sqlsrv` に切り替えるだけで動作する
  ことを目指す。
  - 生SQL（`DB::statement()` 等）やドライバ固有の関数・構文
    （MySQL の `GROUP_CONCAT`、SQL Server の `TOP`/`OFFSET FETCH` 等）を
    直接使わない。Eloquent の標準機能で表現できない場合は
    `design/questions.md` に記録し、両ドライバでの対応方法を検討する。
  - マイグレーションは Laravel の `Schema` ビルダのみで記述し、
    ドライバ固有のカラム型・インデックス構文を直書きしない。
  - **例外（`design/questions.md` Q-17）**: 文字列カラムの型は
    `nvarchar` を使わず `varchar` に統一する方針のため、`$table->string()`/
    `text()`/`char()` が sqlsrv 文法で自動的に `nvarchar` 系へマッピングされて
    しまう分だけは、`DB::statement()` によるマイグレーション後の
    `ALTER COLUMN ... varchar(...)` を許容する（`Modules/Shared/app/Database/
    Support/SqlServerVarchar.php` に集約）。この1点以外で新たなドライバ固有
    分岐・生SQLを追加してはならない。詳細は `design/db-design.md` §5.3 参照。
  - CI・手動テストの両方、または少なくとも定期的に、両ドライバ
    （mysql / sqlsrv）でテストスイートを実行し、ドライバ間の挙動差異を早期に
    検出することが望ましい（プロジェクトの体制に応じて `design/questions.md`
    に運用方針を確認・記録する）。
- **社内の既存システム（基幹システム等）が持つ SQL Server** に直接接続する
  必要がある場合（例: 請求書配信システムの `ShipmentFetchService` が HTTP API
  経由で行っていたような連携を、DB直結に変更する場合）は、Laravel の
  **複数DB接続機能**を用いる。メインDBとは別名の接続として定義し、
  読み取り専用（クエリのみ、書き込み・マイグレーション対象外）として扱う。

### 9.2 `config/database.php` の接続定義（例）

```php
'connections' => [
    'mysql' => [
        // 既定のメイン接続。DB_CONNECTION=mysql の場合に使用
        'driver' => 'mysql',
        // ...
    ],
    'sqlsrv' => [
        // 本番でメインDBをSQL Serverにする場合に使用（DB_CONNECTION=sqlsrv）
        'driver' => 'sqlsrv',
        // ...
    ],
    'legacy' => [
        // 社内既存システム（SQL Server）への読み取り専用接続。
        // メインDBの driver 設定とは独立し、常時 sqlsrv を指す
        'driver' => 'sqlsrv',
        'host' => env('LEGACY_DB_HOST'),
        'database' => env('LEGACY_DB_DATABASE'),
        'username' => env('LEGACY_DB_USERNAME'),
        'password' => env('LEGACY_DB_PASSWORD'),
    ],
],
```

- メインDBの接続名は `DB_CONNECTION` の値（`mysql` または `sqlsrv`）と一致させ、
  `.env` の切り替えだけで済むようにする。
- 既存システムへの接続は `legacy`（プロジェクトの実態に応じて分かりやすい
  名前に変更してよい）のように独立した名前を付け、`.env` も専用の
  `LEGACY_DB_*` 系の変数で管理する（メインDBの `.env` 変数と混同しない）。
- `legacy` 接続を使う Eloquent モデルは `protected $connection = 'legacy';`
  を明示し、`$table` を既存システムの実テーブル名に合わせる。マイグレーション
  対象外（既存システム側のスキーマは Laravel 側で変更しない）。

### 9.3 Docker構成との対応

MySQL コンテナの追加は `docker-compose.yml`・
`docs/システム開発Docker環境設計書.md` の管轄。本書はアプリケーション側の
接続設定方針のみを扱う。DBコンテナを追加した際は同書の「現在の状態」も
更新すること。

## 禁止事項
- 本書が空・未整備のまま実装を進めること
  （`docs/システム開発パイプライン設計書.md` の「src/CLAUDE.md との関係」に
  従い、実装着手前に整備する）
- モジュール型構成を採用せず、Laravel標準のフラットな `app/` 構成に
  全機能を並べること
- 内部利用者（Entra ID認証）と外部利用者（パスワード認証）を同一の
  `users` テーブル・同一ガードで管理すること
- 外部ユーザー向けのMFA・パスキー機能を、選択制ではなく必須として実装し、
  既存ユーザーの利用を妨げること（`design/requirements.md` で必須化が
  明記されている場合を除く）
- WebAuthn／MFAの採用パッケージを、ユーザーに選択肢を提示せず独断で決定すること
- 文字列カラムに `nvarchar` を使用すること（`design/questions.md` Q-17決定）。
  対象は本システムが新規に持つ業務テーブルの文字列カラムで、`varchar` に
  統一する。Laravel フレームワーク標準テーブル（sessions・cache・jobs・
  failed_jobs・password_reset_tokens 等）のデフォルト定義は例外として容認する
  （フレームワークコアのマイグレーションを個別プロジェクトで改変しないため）。
  sqlsrv 接続時に Schema ビルダの `string()` 等が自動的に `nvarchar` へ
  マッピングされてしまう分のみ、マイグレーション末尾での事後 `ALTER COLUMN`
  （`Modules/Shared/app/Database/Support/SqlServerVarchar.php` に集約）を
  唯一の例外として許容する（「9. データベース接続」参照）。
- DBドライバ固有の構文・機能（MySQL固有関数、SQL Server固有のヒント句等）を
  Eloquent／Query Builderを経由せず生クエリに直書きし、ドライバ切り替えで
  動かなくなるコードを書くこと（「9. データベース接続」参照）
