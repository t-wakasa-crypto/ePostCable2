# システム設計パイプライン設計書（design/）

> 本書は [design/CLAUDE.md](../design/CLAUDE.md) から参照される詳細仕様書である。
> `design/CLAUDE.md` は薄いポインタとして維持し、実際の手順・判定条件はすべて
> 本書に記載する。**設計パイプラインの手順・エージェント構成・トリガー・
> 機能追加フローなどに追加・変更を行った場合は、必ず本書を更新すること**
> （`design/CLAUDE.md` 側は参照先が変わらない限り更新不要）。
>
> `design/` 配下のパス表記はすべてプロジェクトルートからの相対パスである
> （作業ディレクトリはルートのまま変わらないため、`design/` 配下に移動しても
> `design/xxx.md` と表記する）。

## セッション開始時の動作

このプロジェクトを開いたら、**ユーザーに話しかける前に必ず現状を確認する**。
何かを尋ねる・提案する前に、以下の Step 1〜3 を省略しないこと。

### Step 1: `design/` と `dev/` の状態確認
`design/` と `dev/` の配下を Glob し、以下の存在有無・内容を確認する。

| ファイル | 確認内容 |
|---|---|
| context.md | 存在するか。「## 未確認事項」が残っていないか（＝初期ヒアリング中断中） |
| questions.md | 存在するか。`→ 保留中` / `→ 決定済み（未反映）` / （状態表記なしの論点）が残っていないか |
| gathered-materials.md 〜 diagrams.md | どこまで作成済みか（パイプラインの到達段階） |
| design/review-document.md | 個別実行系の成果物（レビュー資料）が存在するか |
| dev/implementation-plan.md | 存在するか。タスクのチェックボックス完了率（進捗の詳細な分岐判断は dev 側の設計書が担当するため、ここでは概況の把握のみでよい） |
| feature-log.md | 存在するか。「ヒアリング中」「影響範囲判定済み・未反映」のエントリが残っていないか（＝機能追加フローが中断中・反映待ち） |

### Step 2: 状態に応じた分岐
- **`design/` が空 or 存在しない** → 初回起動。次のメッセージを送る：
  「設計パイプラインへようこそ。どんなシステムを作りたいですか？概要を教えてください。一緒にヒアリングしながら設計を進めます。」
- **context.md に未確認事項が残っている** → 初期ヒアリングが中断中。何が確定済みで
  何が未確認かを要約し、その続きから質問を再開する。
- **questions.md に `保留中` / `決定済み（未反映）` / 未解消の論点がある** → 論点解消
  ヒアリングが中断中、または反映待ち。該当件数と内容を要約し、
  「決定済み（未反映）」があれば `questions-resolver` の起動を提案し、
  「保留中」「未解消」があればヒアリングの続きを提案する。
- **feature-log.md に「ヒアリング中」「影響範囲判定済み・未反映」のエントリがある** →
  機能追加フローが中断中、または反映待ち。該当エントリを要約し、
  「影響範囲判定済み・未反映」があれば該当エージェントの起動を提案し、
  「ヒアリング中」があればヒアリングの続きを提案する。
- **上記いずれも該当せず、パイプラインが完了段階まで進んでいる** →
  どの成果物（gathered-materials.md 〜 diagrams.md、dev/implementation-plan.md、
  design/review-document.md）が揃っているかを一覧で示し、次に取りうる行動
  （個別実行系の起動、機能追加、追加資料の反映、レビュー等）を提案する。
  加えて、`detailed-design.md` / `db-design.md` / `diagrams.md` が
  すべて揃っている（＝設計パイプライン本流が完了している）場合は、必ず
  「まず `./scripts/setup.sh` を実行して開発環境を構築し、完了したら
  『開発パイプラインを開始して』と伝えてください」と明示的に案内する
  （`design/`・`dev/`・`src/` は同一プロジェクト内で共存するため、コピーや
  引き渡しは不要。他の提案の一つとして埋没させず、独立した行として伝える）。

### Step 3: 状況報告
分岐の結果に関わらず、ユーザーへの最初の発言には必ず次を含める：
1. 現在どこまで進んでいるか（一覧または一言サマリー）
2. 次に何をする必要があるか・何を選べるか（提案は1〜3個に絞る）

「ようこそ」の定型文だけを送って終える、または前回の続きを何も確認せずに
新規ヒアリングを始める、といった動作は禁止する。

## 起動トリガー
以下のような発言があった場合も設計パイプラインの開始と判断してヒアリングを始める：
- 「設計して」「設計したい」
- 「〇〇を作りたい」「〇〇システムを考えたい」
- 「要件定義して」「仕様を考えて」
- 「アプリを作りたい」「サービスを作りたい」

### トリガー衝突時の確認（`design/requirements.md` が既に存在する場合）
`design/requirements.md` が既に存在する状態で、上記の**新規設計トリガー**
（機能追加トリガーではない方）が発話された場合、意図が「機能追加」なのか
「別プロジェクトとしてゼロから作り直したい」なのか判別できない。
勝手にどちらかを選ばず、`AskUserQuestion` で意図を確認してから動くこと。
無確認で `requirements-gatherer` 以降を再実行し、既存の確定事項を
上書き・喪失させることは禁止する。

### 機能追加トリガー（`design/requirements.md` が既に存在する場合）
`design/requirements.md` が存在する状態（＝一度パイプラインが完了済み）で
以下のような発言があった場合は、新規設計ヒアリングではなく
「機能追加フロー」（後述）のヒアリングを開始する：
- 「機能追加したい」「〇〇機能を追加したい」「〇〇機能を足したい」
- 「この機能も欲しい」「〇〇できるようにしたい」
- 「〇〇画面を追加したい」「〇〇の一覧画面が欲しい」（画面追加）
- 「〇〇テーブルを追加したい」「〇〇の項目を持たせたい」（データ構造の追加・変更）
- 「〇〇ボタンを追加したい」「〇〇できるボタンが欲しい」（軽微なUI追加）

これらは規模の大小に関わらず**すべて同じ機能追加フローに乗せる**。規模に応じて
「影響範囲の判定」で対象外の設計ファイルを「不要」とマークすればよく
（例: ボタン追加なら requirements.md 以外は不要、テーブル追加なら
basic-design.md は不要、というように自然に軽量化される）、規模ごとに別の
フローを用意する必要はない。

`design/requirements.md` がまだ存在しない場合は、上記の発言でも通常の新規設計
ヒアリングとして扱う。

### 機能追加フローの対象外（アドホック修正）
設計内容の変更を伴わない直接的なコード修正（バグ修正・軽微な文言修正・
「動かないので直して」等）は、設計変更ではないため機能追加フローの対象外と
する。この種の修正は `docs/システム開発パイプライン設計書.md`「アドホック
修正フロー」に従い、design/ を経由せず直接 `implementer` へ渡す。判断に迷う
場合（修正が要件・仕様の変更を伴うかどうか不明な場合）は、`AskUserQuestion`
でユーザーに確認する。

## パイプライン全体像

```
【設計パイプライン / このプロジェクト】─────────── 半自動連鎖（各段でユーザー確認あり）
メインチャット（ヒアリング）
  → design/context.md
  → requirements-gatherer  → gathered-materials.md
  ⏸ 確認ポイント1（既存）: 材料の過不足をユーザーと確認しブラッシュアップ
  → requirements-analyst   → requirements.md
  ⏸ 確認ポイント2（必須）: スコープ・FR一覧の要点を提示し、続行可否を確認
  → basic-designer         → basic-design.md
  ⏸ 確認ポイント3（必須）: 構成・技術選定の要点を提示し、続行可否を確認
  → detailed-designer      → detailed-design.md
  → db-designer            → db-design.md          ┐ 並列（詳細設計確認後は自動）
  → diagram-designer       → diagrams.md           ┘
  ⏹ パイプライン停止・完了報告

【個別実行 / このプロジェクト】────────────────── 手動起動
  「実装計画を作って」     → implementation-planner → dev/implementation-plan.md
  「レビュー資料を作って」  → review-doc-creator    → design/review-document.md
  「論点を反映して」       → questions-resolver     → questions.md + 関連する全設計ファイル
                            （メインチャットが対話で論点を決定 → questions.md に
                              決定済み・未反映として記録 → questions-resolver 起動、の順）

【機能追加フロー / このプロジェクト】──────────── 手動起動（requirements.md 確定後のみ）
  「機能追加したい」
  → メインチャット（機能追加ヒアリング）
  → design/feature-log.md（機能名・背景・スコープ・影響範囲を記録）
  → 影響範囲に該当するエージェントのみ「追記モード」で起動
     requirements-analyst → basic-designer → detailed-designer
     → db-designer / diagram-designer（並列、対象の場合のみ）
  → 各エージェントが feature-log.md の反映状況を「反映済み」に更新
  ⏹ 全対象が反映済みになったら完了報告

【開発パイプライン / 同一プロジェクト内】──────── 全自動連鎖（Claude Code による完全自律実装。
                                                          ユーザー確認なしで下記を回し続ける）
  design/・dev/・src/ は同一プロジェクト（同一リポジトリ）内に共存するため、
  ファイルのコピー・引き渡し・再同期は不要（旧: WSL への引き渡し・再同期ルールは廃止）。
  実装作業は implementer エージェントが src/ 配下で自律的に行う。進捗管理・
  レビューループの詳細は docs/システム開発パイプライン設計書.md を参照する。
  「開発パイプラインを開始して」（または「実装して」）
  ⏸ 前提条件: 開発環境の確認（scripts/setup.sh による構築が完了しているか。
     詳細は docs/システム開発パイプライン設計書.md「開発環境の確認」章を参照）。
     未整備なら先に setup.sh の実行を案内し、implementer は起動しない
     （唯一の起動前提条件のブロック）。
  → implementer            → src/ 配下のコード + dev/implementation-plan.md（チェック更新）
  → code-reviewer          → dev/review-report.md（総合評価: OK/要修正/差し戻し）
  → 要修正・差し戻しなら確認なしで implementer を再起動（修正モード）→ 再レビュー
     （OK になるまで自動で繰り返す。ただし3往復してもOKにならない場合は
       自動ループを停止しユーザーに報告する。詳細は
       docs/システム開発パイプライン設計書.md「自律ループの継続条件・停止条件」を参照）
```

> **同期に関する注意**: 上記「設計パイプライン」の順序・遷移は
> `.claude/settings.json` の `SubagentStop` フックが表示する案内文（次に
> どのエージェントを起動するか）と重複して定義されている。パイプラインの
> 順序やエージェント構成を変更した場合は、必ず `.claude/settings.json` の
> 該当するフック文言も合わせて更新すること。

## 出力ファイル規則
設計フェーズの成果物は `design/` フォルダに、開発フェーズの進行管理成果物
（実装計画・レビュー結果）は `dev/` フォルダに保存すること。実装コードのみ
例外的に `implementer` が `src/` フォルダへ出力する。上記以外の場所に
成果物を保存しない。

| エージェント | 入力ファイル | 出力ファイル |
|---|---|---|
| requirements-gatherer | design/context.md（任意） | design/gathered-materials.md |
| requirements-analyst | design/gathered-materials.md | design/requirements.md |
| basic-designer | design/requirements.md | design/basic-design.md |
| detailed-designer | design/basic-design.md | design/detailed-design.md |
| db-designer | design/detailed-design.md + design/requirements.md（NFR確認用） | design/db-design.md |
| diagram-designer | design/detailed-design.md + design/basic-design.md（全体像確認用） | design/diagrams.md |
| implementation-planner | design/ 全ファイル + 開発環境ファイル（`dev-environment.md` 等、存在する場合のみ・任意） | dev/implementation-plan.md |
| review-doc-creator | design/ 設計ファイル | design/review-document.md |
| implementer | design/ 全ファイル + dev/implementation-plan.md + src/CLAUDE.md + dev/review-report.md（修正モード時） | src/ 配下の実装コード + dev/implementation-plan.md（チェック更新） |
| code-reviewer | design/ 全ファイル + dev/implementation-plan.md + src/ の実装コード | dev/review-report.md |
| questions-resolver | design/questions.md（決定済み・未反映の論点） | design/questions.md + 関連する全設計ファイル（上書き更新） |
| requirements-analyst 〜 diagram-designer（追記モード時） | design/feature-log.md（影響範囲判定済み・未反映のエントリ）+ 自分の既存出力ファイル | 自分の出力ファイル（追記編集）+ feature-log.md（反映状況更新） |

## メインチャットの役割（対話型オーケストレーター）

パイプラインを開始する前に、メインチャット（Claude）がユーザーとの対話で情報を収集する。
収集した情報は `design/context.md` に保存してから gatherer を起動すること。

### ヒアリング終了（gatherer 起動）の判断基準
以下がすべて埋まったら「情報が十分集まった」と判断し、gatherer を起動してよい。
埋まっていない項目があるまま自己判断で打ち切らないこと。
- システム概要（何を・誰が・なぜ）が一通り言語化されている
- 技術スタック等の既知の制約が確認済み、または「制約なし・任せる」と明言されている
- 競合調査の要否が明言されている
- MVP スコープ（含む／除外）の大枠が示されている
一部が曖昧なまま先に進みたいとユーザーが明言した場合は、その旨を
`design/context.md` の「## 未確認事項」に残してから gatherer を起動してよい。

### context.md に記載する内容
```markdown
# 対話で収集した情報

## システム概要
（何を作るか・誰が使うか・なぜ作るか）

## 既知の制約・前提
（技術スタック指定・予算・期限・法規制など）

## 競合調査の要否
（必要 / 不要 / 対象サービス名）

## 優先度・スコープ
（MVP に含める機能・除外する機能）

## その他メモ
（ユーザーが言及したこと）
```

### gatherer 起動後のブラッシュアップ
gatherer が生成した `design/gathered-materials.md` をメインチャットが確認し、
不足・誤りがあればユーザーに確認して `design/context.md` を更新し、
gatherer を再起動するか、直接 `design/gathered-materials.md` を編集してから
次のエージェントへ進む。

### 確認ポイント2・3（requirements-analyst 後 / basic-designer 後）
`requirements-analyst` と `basic-designer` は後続工程が積み上げる土台であり、
ここでの誤りは detailed-design / db-design / diagrams まで無自覚に伝播する。
そのため以下の2箇所は**自動で次のエージェントを起動せず、必ず一度止まる**。

1. `requirements-analyst` 完了後: スコープ・主要 FR の一覧・questions.md に
   記録された論点の件数を3〜5行で要約し、「この内容で基本設計に進めてよいか」
   をユーザーに確認する。訂正があれば requirements.md を修正してから再確認する。
2. `basic-designer` 完了後: システム構成・技術スタック選定の要点を要約し、
   「この内容で詳細設計に進めてよいか」を確認する。

承認が得られたら次のエージェントを起動する。`detailed-designer` 完了後は
db-designer・diagram-designer が並列で自動起動してよい（両者は独立した
出力ファイルに書き込むため、詳細設計の承認をもって進めてよい）。

## エージェント連鎖ルール
- 各エージェントは必ず上記の入力ファイルを最初に読み込むこと
- 矛盾・不明点を発見した場合は `design/questions.md` に記録し、上流ファイルを修正してから進むこと
- 出力ファイルへの保存が完了したら「[ファイル名] に保存しました」と報告すること
- db-designer と diagram-designer は detailed-designer 完了後に並列で起動すること
- db-designer・diagram-designer 完了後は自動連鎖せず停止すること

## 論点解消フロー（questions.md の反映）

各サブエージェントが記録した `design/questions.md` の論点は、後から
メインチャットとユーザーの対話で解消されることが多い。解消しただけで終わらせず、
以下の手順で関連ファイルへの反映まで必ず行う。

1. メインチャットがユーザーと対話し、論点ごとに決定を得る（AskUserQuestion 等を使用）。
2. 決定が出た論点は、都度 `design/questions.md` の該当箇所に決定内容を書き込み、
   ステータスを `→ 決定済み（未反映）` にする（他ファイルへの反映はまだ行わない）。
3. 1論点ずつ反映すると手戻り・反映漏れが起きやすいため、ある程度まとまった単位
   （優先度グループ単位、または「これで最後」と判断できた時点）で `questions-resolver`
   を起動し、`design/` 配下の関連ファイルへ一括反映させる。
4. `questions-resolver` の報告を確認し、反映漏れがあれば追加修正するか再度エージェントを起動する。
5. 反映完了後、`dev/implementation-plan.md` や `design/review-document.md` が
   既に存在する場合は、内容が古くなっている可能性をユーザーに伝える
   （詳細は「個別実行系成果物の陳腐化」節を参照）。

> メインチャット自身が反映作業まで行ってもよいが、対象ファイルが多い場合は
> `questions-resolver` に委譲し、反映漏れを構造的に防ぐこと。

## ヒアリング中断・再開ルール

ヒアリング（初期ヒアリング・論点解消ヒアリング・機能追加ヒアリングいずれも）は
ユーザーの都合でいつでも中断されうる。「ここで一旦やめる」「続きは後で」「また今度」
等の発言があった場合:

1. その時点までに得られた回答を、対応するファイルに必ず保存する（口頭の回答を
   会話内にとどめず、ファイルに永続化する）。
   - 初期ヒアリング中 → `design/context.md` に確定済み項目を反映し、
     未確認の項目は「## 未確認事項」として残す。
   - 論点解消ヒアリング中 → 回答済みの論点は `design/questions.md` に
     `→ 決定済み（未反映）` として書き込み、未回答の論点はステータスを
     `→ 保留中`（着手したが未回答）としてマークする。
   - 機能追加ヒアリング中 → `design/feature-log.md` の該当エントリに、
     判明済みの項目（機能名・背景・要望内容・スコープ・影響範囲）を書き込めるだけ
     書き込み、ステータスは `ヒアリング中` のまま残す（影響範囲が未判定なら
     判定表は空欄のままでよい）。
2. 何が保存され、何が残っているかをユーザーに一言で報告する
   （例:「Q-01〜Q-05まで確定しました。Q-06以降は保留のままです」
   「FA-02は背景・要望内容までは記録済み、影響範囲の判定はまだです」）。
3. 途中まで決定済みの論点があれば、中断前でも `questions-resolver` を起動して
   反映してよい（未回答分を待つ必要はない）。機能追加も同様に、影響範囲判定まで
   終わっていれば中断前に対象エージェントを起動してよい。
4. 再開時（「続き」「さっきの続きから」等）は、該当ファイルの `保留中` / 未確認事項 /
   `ヒアリング中` のエントリを読み込み、そこから再開する。何を確認済み・未確認かを
   再度要約してから質問を再開する。

## 資料追加ルール

パイプライン進行中・完了後に、ユーザーが新しい資料（追加ドキュメント・URL・
思い出した制約など）を提示することがある。現在の進行状況に応じて扱いを分ける。

1. **gathered-materials.md 作成前**: `design/context.md` に追記し、通常どおり
   `requirements-gatherer` を起動する。
2. **requirements.md 未作成・gathered-materials.md 作成済み**: `design/context.md`
   に追記した上で `design/gathered-materials.md` にも反映するか、
   `requirements-gatherer` を再起動する。
3. **requirements.md 以降が確定済み**: 新資料の内容を勝手に既存の確定事項へ
   上書きしない。まず新資料が既存の設計と矛盾しないか確認し、
   - 矛盾がなく単なる追加情報 → 影響する設計ファイルに直接追記する
     （軽微な追記は questions.md を経由しなくてよい）。
   - 既存の確定事項と矛盾する、または重要な設計判断に関わる
     → `design/questions.md` に新規論点として記録し、ユーザーに確認してから
     （論点解消フローに沿って）反映する。
   - 資料の出典（ファイル名・URL・受領日）は `design/context.md` の
     「その他メモ」に必ず記録し、後から追跡できるようにする。
4. どのケースでも、資料そのもの（PDF・元ファイル）は `design/` 配下にコピーせず、
   要点を Markdown へ要約して既存ファイルに統合する（成果物は `design/` の
   Markdown ファイル群に一本化する方針を維持するため）。

## 機能追加フロー（requirements.md 確定後の新機能追加）

### 背景
requirements-analyst 〜 diagram-designer の各エージェントは「単一の上流ファイルを
読んで出力ファイルを新規生成する」設計になっている。パイプライン完了後に機能追加が
発生した際、単純に `context.md` へ追記して `requirements-analyst` 以降を再実行すると、
過去のヒアリングで積み上げた決定事項（論点解消フローで反映した内容など、上流ファイルには
現れず出力ファイルにしか残っていない情報）が全体再生成によって失われる。
これを避けるため、機能追加は既存ファイルへの**追記編集**として扱う。

### メインチャットの手順

1. **ヒアリング**: 機能追加トリガーを検知したら、以下を対話で確認する。
   - 機能名・背景（なぜ必要か）
   - 要望内容（何をできるようにしたいか）
   - スコープ（含む／含まない）
   - 優先度・既存機能への影響有無
2. **feature-log.md への記録**: `design/feature-log.md` が存在しなければ新規作成し、
   新しいエントリを追記する（形式は下記）。この時点のステータスは「ヒアリング中」。
3. **既存の確定事項との矛盾チェック（必須）**: 影響範囲を判定する前に、
   `design/questions.md`（解消済みの決定事項）と `design/requirements.md` の
   スコープ外事項（例:「〇〇機能は現在・将来ともに提供しない」等の明記）を確認する。
   新機能の要望がこれらと矛盾する場合は、影響範囲判定に進まず、矛盾内容をユーザーに
   提示して「過去の決定を覆してよいか」を確認する。覆す場合は questions.md の
   該当箇所に「2026-MM-DD の機能追加 FA-xx により決定を変更」と追記してから進める。
4. **影響範囲の判定**: 機能内容から、requirements.md / basic-design.md /
   detailed-design.md / db-design.md / diagrams.md のどれが影響を受けるかを判断する。
   判断が難しい場合は `AskUserQuestion` でユーザーに確認してよい。
   判定結果を feature-log.md の表に記入し、ステータスを「影響範囲判定済み・未反映」にする。
   - 一般的に、新しい機能は detailed-design.md（API・ロジック）には必ず影響する。
   - データを保持する機能なら db-design.md も対象。
   - 業務フロー・画面遷移が変わるなら diagrams.md も対象。
   - 既存のシステム構成・技術選定を変えない限り basic-design.md は対象外でよい。
   - **requirements.md は原則「要」とする**。既存の FR で完全に説明できる
     軽微な挙動追加である場合に限り「不要」にしてよい。新しい振る舞いを
     どの FR にも紐づけられない場合は、必ず新規 FR を追加すること
     （対応する FR のない設計変更を basic-design 以降に発生させないため）。
5. **対象エージェントの起動**: 「要」とマークしたファイルに対応するエージェントを、
   通常のパイプライン順（requirements-analyst → basic-designer → detailed-designer
   → db-designer / diagram-designer 並列）で起動する。「不要」の段は起動しない
   （そのエージェントの出力ファイルは前段の更新をそのまま入力として使われるだけで変更されない）。
   各エージェントは feature-log.md を見て自動的に追記モードで動作する（Step 0 参照）。
6. **完了確認**: 全対象行が「反映済み」になったら、エントリ全体の「状態」を
   「設計反映済み」に、「開発への反映」を「未反映」に更新し、変更概要を
   ユーザーに報告する。`dev/` パイプラインが次に起動された際（`implementer`
   起動前の状態確認、または `implementation-planner` の起動時）、この
   「開発への反映: 未反映」エントリが自動検知され、`implementation-planner`
   が追記モードで `dev/implementation-plan.md` にタスクを追記する（ユーザーが
   別途「実装計画を作って」等を言う必要はない。詳細は
   docs/システム開発パイプライン設計書.md「機能追加の自動反映」章を参照）。
   `design/review-document.md` が既に存在する場合は、内容が古くなっている旨も
   併せて伝える（詳細は「個別実行系成果物の陳腐化」節を参照。こちらは自動
   反映されないため引き続きユーザー判断で再生成する）。

### feature-log.md の形式

```markdown
# 機能追加ログ

## FA-01: [機能名]（記録日: YYYY-MM-DD）

**状態**: ヒアリング中 / 影響範囲判定済み・未反映 / 設計反映済み
**開発への反映**: 未反映 / 反映済み（実装計画へ追記済み） / 実装完了
  （最初の2つの状態のときは常に「未反映」。「設計反映済み」になって
  初めて「未反映」→「反映済み」の遷移対象になる。「実装完了」は
  `implementer` が対応タスクをすべて完了しチェックした時点で
  `implementer` 自身が更新する）

**背景・目的**: ...
**要望内容**: ...
**スコープ（含む/含まない）**: ...

**影響範囲**:
| ファイル | 要否 | 反映状況 |
|---|---|---|
| requirements.md | 要/不要 | 未反映/反映済み |
| basic-design.md | 要/不要 | 未反映/反映済み |
| detailed-design.md | 要/不要 | 未反映/反映済み |
| db-design.md | 要/不要 | 未反映/反映済み |
| diagrams.md | 要/不要 | 未反映/反映済み |

**反映後の変更概要**: （全対象が反映済みになったら記載）
```

過去の FA エントリは削除せず、機能追加の履歴として積み上げていく
（questions.md と同様、追記のみで運用する）。

## 個別実行系成果物の陳腐化

`dev/implementation-plan.md`・`design/review-document.md`・`dev/review-report.md` は
生成された時点の設計内容をもとにしたスナップショットであり、以降に
論点解消フロー・機能追加フローで設計ファイルが更新されても自動的には追従しない。

- 論点解消フロー・機能追加フローの完了報告時、これらのファイルが存在するなら
  「設計が更新されたため内容が古い可能性がある」旨を必ず伝える。
- 再生成するかはユーザーの判断に委ねる（自動では再生成しない。手戻りコストは
  ユーザーが状況に応じて判断すべきため）。
- 再生成する場合は、対応する個別実行エージェント（implementation-planner /
  review-doc-creator / code-reviewer）を通常どおり起動する（追記モードはない。
  これらは元々「最新の設計一式」から都度生成するファイルのため、上書き生成でよい）。

## 再同期ルールについて（廃止・統合済み）

design/・dev/・src/ が同一プロジェクト内に共存する構成に変更したため、
かつての「WSL への引き渡し・再同期ルール」は不要になった。論点解消・機能追加で
design/ 配下が更新された場合、dev/・src/ 側の作業者（＝同じセッションの
メインチャット）はそのまま最新の design/ を参照できる。
陳腐化しうるのは生成物（dev/implementation-plan.md 等）のみであり、
その扱いは前節「個別実行系成果物の陳腐化」に従う。

なお `.claude/agents/`・`.claude/skills/` および `CLAUDE.md` 群は
`claude-pipeline-template` リポジトリ由来のフレームワーク部分であり、
その更新反映範囲は `.claude/SYNC_MANIFEST` を参照すること
（プロジェクト固有の成果物とは明確に区別する）。

## 禁止事項
- `design/`・`dev/` 以外への成果物の保存
- 開発フェーズの成果物（実装計画・レビュー結果）を `design/` に保存すること
  （必ず `dev/` に保存する）
- 入力ファイルを読まずに処理を開始すること
- 未確認の前提での設計判断（必ず questions.md に記録）
- ヒアリングの回答をファイルに保存しないまま中断・終了すること
- questions.md で決定済みの論点を、関連ファイルへ未反映のまま放置すること
- 機能追加を、既存の確定済み設計ファイルへの全体再生成（Write による上書き）で反映すること
  （必ず feature-log.md 経由の追記モード＝Edit による差分反映を用いる）
- 確認ポイント2・3（requirements-analyst 後・basic-designer 後）をスキップして
  次のエージェントを自動起動すること
- `requirements.md` が既存の状態で、意図確認（AskUserQuestion）なしに
  新規設計トリガーを新規プロジェクトとして処理すること
- 機能追加の要望が questions.md の既存決定と矛盾するのに、確認なく上書き・
  無視して設計に反映すること
