## 1. バッチ・定期実行ジョブの一覧と対処

スケジュール定義の詳細は `design/detailed-design.md` §1.1.4 / `design/requirements.md` FR-04 を参照。

| 実行時刻 | コマンド | 役割 | 失敗時の対処 |
|---|---|---|---|
| 毎日 12:15〜12:30 | `batch:fetch-shipment-data` | 基幹から出荷データ取得・書類作成 | 手動起動で救済（出荷取得履歴画面の「バッチ手動起動」/ admin）。基幹の請求データ確定は翌日12:00のため、確定遅延時は時刻を後ろ倒しして手動実行 |
| 毎日 01:00 | `batch:send-delivery-notes` | 納品書メール送信バッチ | 手動起動（納品書一覧の「バッチ手動起動」/ admin）。残留（processing）は次回起動時に stuck-timeout で自動差し戻し |
| 毎日 01:30 | `batch:send-invoices` | 請求書メール送信バッチ | 同上（請求書一覧の「バッチ手動起動」/ admin） |
| 毎週月曜 02:00 | `batch:send-delivery-notes --retry-failed` | failed 納品書の再送 | 個別是正は納品書詳細の手動再送、または一括再キュー（admin） |
| 毎週月曜 02:30 | `batch:send-invoices --retry-failed` | failed 請求書の再送 | 同上 |

- 全バッチは `Cache::lock(key, 3600)` で二重起動を防止する（NFR-R-01）。手動起動が「何も起きない」場合は、既に同一バッチが実行中でロック未取得の可能性がある（`storage/logs/laravel.log` の「ロック取得失敗のためスキップ」を確認）。
- バッチ完了通知は `admin_notification_emails`（システム設定画面）設定時に `BatchSummaryMail` で届く（NFR-R-07）。未設定時は警告ログのみ。

## 2. ログ確認・目視監視方針（NFR-M-06 / NFR-M-07）

- **監視方針**: 専用の監視・アラートツールは導入しない。アプリログ（Laravel Log）を**目視で定期確認**する運用とする（NFR-M-07）。
- **主なログ確認先**:
  - アプリログ: `src/storage/logs/laravel.log`（コンテナ内 `/var/www/html/storage/logs/laravel.log`）。
    - `docker compose exec php tail -f storage/logs/laravel.log` で追尾できる。
  - バッチ実行履歴: 出荷取得は `shipment_fetch_logs`（出荷取得履歴画面）、メール送信は `send_mail_logs` / `send_mail_log_items`（メール送信履歴画面）で追跡する（NFR-M-02）。
  - キュー失敗ジョブ: `failed_jobs` テーブル（`php artisan queue:failed`）。
- **重点確認ポイント**:
  - `ShipmentFetchLog.status = failed`（基幹 API 接続失敗・整合性エラー）。`error_message` を確認。
  - 書類 `status = failed_permanent`（送付先0件・無効アドレス）。`updateEmails`（admin）でアドレス是正後、一括再キューで復旧（`design/requirements.md` FR-08 / BR-07）。
  - 明細合計と `amount` の不一致でエラーマーキングされた書類（ログの「明細合計と amount が不一致」警告）。データ修正は当面 DB 直接編集で対応（DB-Q-02）。
- **ログ保存期間**: 7年間（NFR-M-06）。ローテーション設定・長期保管（アーカイブ）はインフラ運用側で担保すること。`config/logging.php` の `daily` チャンネル `days` はデフォルト14日のため、7年保管はログ収集基盤またはファイルアーカイブで対応する（アプリ側でのファイル保持のみに依存しない）。

## 3. PDF 保存パスと確認方法

PDF 生成仕様は `design/detailed-design.md` §1.2 / §1.3.2、`design/requirements.md` FR-15 を参照。

- **保存先（バッチ送信時のみ保存・NFR-P-05）**:
  - 請求書: `storage/app/private/invoices/{年}/{月}/invoice_{invoice_number}.pdf`（年月基準は `issue_date`）。
  - 納品書: `storage/app/private/delivery-notes/{年}/{月}/delivery_{delivery_number}.pdf`（年月基準は `delivery_date`・OQ-11）。
  - ※ 実パスは `config/filesystems.php` の `local` ディスク root に従う。
- **画面ダウンロード**: Storage に PDF があればそれを返却、なければ即時生成する（この即時生成分は保存しない・NFR-P-05）。
- **確認方法**: `docker compose exec php ls -R storage/app/private/invoices`。生成失敗（空出力）は `RuntimeException` となりジョブが `failed` 遷移する（`send_mail_log_items.error_message` を確認）。

## 4. SQL Server 運用上の留意点（db-design §6.3）

本番 DB は外部 SQL Server（`db-sv03.solid-corp.local:1433` / DB 名 `ePostCable`・BR-10）。詳細は `design/db-design.md` §6.3 を正本とする。運用側での実施事項は以下。

- **統計情報の更新**: 大量バッチ投入後（出荷取得バッチ等）にクエリプランが劣化しうる。定期的に `UPDATE STATISTICS` を実行する運用計画を持つこと。
- **インデックスメンテナンス**: 大量 UPDATE（`status` 変更）でインデックス断片化が進む。定期的に `ALTER INDEX ... REBUILD` / `REORGANIZE` を実施する。対象は送信待ち取得の複合インデックス `(status, created_at)` および `status` 単独インデックス（`invoices` / `delivery_notes`・NFR-P-03）を重点とする。
- **外部接続依存**: DB コンテナを持たず外部 SQL Server に全依存。DB 障害・ネットワーク断で全機能停止するため、接続監視・接続タイムアウト設定は運用設計（インフラ側）で対応する。
- **トランザクション分離レベル**: 送信バッチは `lockForUpdate()` で行ロックする（詳細設計 §1.1.2 / NFR-P-04）。スナップショット分離（RCSI）を有効化している場合は `UPDLOCK` ヒント相当の挙動を事前確認すること。

### DB ドライバ切替（mysql / sqlsrv）

- アプリは Eloquent / Schema ビルダのみで実装し、ドライバ非依存を維持している（開発環境設計書 §9 / RK-8）。`.env` の `DB_CONNECTION` を `mysql` ⇄ `sqlsrv` に切り替えるだけで動作する想定。
- ドライバ非依存性の回帰は `Modules/Shared/tests/Feature/DatabaseDriverPortabilityTest.php` で静的に担保している（生 SQL・エンジン/照合順序指定の混入を検出）。
- ※ 本番 SQL Server は開発環境から到達不可のため、実インスタンスに対するテストスイート実行は本番/検証環境での実施とする。

## 5. トラブルシューティング（よくある事象）

| 事象 | 想定原因 | 対処 |
|---|---|---|
| メールが届かない（status=failed_permanent） | 送付先0件・無効アドレス（BR-01/BR-04） | `updateEmails`（admin）でアドレス是正 → 一括再キュー |
| メールが届かない（status=failed） | SMTP 一時障害・PDF 生成失敗等 | 自動リトライ（`max_retries` 回・`retry_backoff` 間隔）後も failed なら `--retry-failed` / 一括再キュー |
| 書類が processing のまま滞留 | ワーカークラッシュ・予期せぬタイムアウト | 次回送信バッチ起動時に stuck-timeout（既定60分）で pending へ自動差し戻し（NFR-R-04）。即時復旧はバッチ手動起動 |
| バッチ手動起動しても動かない | 同一バッチ実行中でロック未取得 | ログの「ロック取得失敗のためスキップ」を確認。完了を待って再実行 |
| 設定変更が反映されない | 反映は次回ジョブ生成時（NFR-M-04） | ワーカー再起動は不要。実行中ジョブには適用されない点に留意 |
| テスト実行時に `.temp/test-results` の書き込み権限警告 | Pest の結果キャッシュ書き込み先（`vendor/pestphp/pest/.temp`）の権限 | テスト結果自体には影響しない軽微な警告。抑止する場合は該当ディレクトリの書き込み権限を付与するか、CI ではキャッシュを無効化して実行する |
