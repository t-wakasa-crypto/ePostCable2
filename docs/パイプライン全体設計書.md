# claude-pipeline-template パイプライン全体設計書

> 本書は [claude-pipeline-template/CLAUDE.md](../CLAUDE.md)（ルーター）から
> 参照される、`claude-pipeline-template` そのもの（設計パイプライン・開発
> パイプラインの仕組み）の設計思想・全体構成をまとめた見取り図である。
> `docs/CLAUDE.md` の本来の役割（実装済みシステムの運用ドキュメント）とは
> 対象が異なるが、プロジェクト構成を理解するための参照資料として `docs/`
> 配下に置く。
>
> 各領域の詳細な手順・判定条件の正本は、本書ではなく「9. 詳細設計書
> 一覧」に挙げる個別の設計書（`docs/システム設計パイプライン設計書.md` 等）
> である。本書はそれらの要約・見取り図であり、矛盾があれば個別の設計書側を
> 正としてこの文書を更新する。**パイプライン全体の構成・変更経路・既知の
> 前提に追加・変更を行った場合は、必ず本書を更新すること**
> （[claude-pipeline-template/CLAUDE.md](../CLAUDE.md) の「更新ルール」参照）。

## 1. 全体構成

このプロジェクトは、Linux（WSL）上で動作する Laravel 13 の開発環境のひな型
（gitリポジトリ）であり、6つの領域を同一プロジェクト内のフォルダで管理する。

| フォルダ | 領域 | 内容 |
|---|---|---|
| `design/` | システム設計パイプライン | ヒアリング・要件定義・基本/詳細設計・DB設計・図解・論点解消・機能追加 |
| `dev/` | システム開発パイプライン | 実装計画・進捗管理・コードレビューループ・進行管理（自律実装） |
| `src/` | システム開発環境 | Laravel 13 の開発プロジェクトルート（`composer.json`・`artisan`・`app/` 等） |
| `docker/` | docker環境 | Dockerfileなどのdocker環境 |
| `docs/` | システム開発ドキュメント | Laravel 13 プロジェクトの開発ドキュメント（本書もここに置く） |
| `scripts/` | システム開発スクリプト | 開発環境構築・docker操作・artisan/composer操作等の補助スクリプト |

`design/`・`dev/` は専用のサブエージェント（`.claude/agents/`）が動く
**パイプライン領域**。`src/`・`docker/`・`docs/`・`scripts/` は専用エージェントを
持たない**参照・実体領域**（`src/` への実装コード書き込みだけは `dev/` パイプラインの
`implementer` が担う）。

## 2. 使い方の全体フロー

1. git から `claude-pipeline-template` をクローンする
2. フォルダ名をプロジェクト名に変更する
3. プロジェクトフォルダで設計パイプラインを実行し、ヒアリングからシステム仕様を作成する
4. `./scripts/setup.sh` を実行して開発環境を構築する
5. 「開発パイプラインを開始して」と伝えると、`implementer`↔`code-reviewer` が
   完全自律で回り続け、Laravel 実装が完成する
6. 完成後の変更（機能追加・画面追加・テーブル追加・ボタン追加・バグ修正等）は
   後述の「6. 完成後の変更経路」に従う

## 3. 設計パイプライン（design/）

詳細は [システム設計パイプライン設計書.md](システム設計パイプライン設計書.md)
（[design/CLAUDE.md](../design/CLAUDE.md) が参照する設計書）を参照。
ヒアリング → 各段階の設計エージェントへの本流連鎖と、確認ゲートで構成される。

```
メインチャット（ヒアリング） → context.md
  → requirements-gatherer  → gathered-materials.md
  ⏸ 確認ポイント1: 材料の過不足を確認
  → requirements-analyst   → requirements.md
  ⏸ 確認ポイント2（必須）: スコープ・FR一覧を提示し続行可否確認
  → basic-designer         → basic-design.md
  ⏸ 確認ポイント3（必須）: 構成・技術選定を提示し続行可否確認
  → detailed-designer      → detailed-design.md
  → db-designer            → db-design.md   ┐ 並列
  → diagram-designer       → diagrams.md    ┘
  ⏹ 完了。次に scripts/setup.sh の実行と開発パイプラインの開始を案内
```

セッション開始時は必ず `context.md`・`questions.md`・`feature-log.md` 等の状態を
確認し、ヒアリング中断や論点未解消があれば自動的に続きを再開する（詳細は
設計書の「セッション開始時の動作」）。

## 4. 開発パイプライン（dev/）— 完全自律開発

詳細は [システム開発パイプライン設計書.md](システム開発パイプライン設計書.md)
（[dev/CLAUDE.md](../dev/CLAUDE.md) が参照する設計書）を参照。design で確定した
仕様をもとに、Claude Code が**ユーザーに逐一確認を取らずに**実装を完了させる。

```
implementation-planner（実装計画の生成。未生成なら自動で最初に起動）
  → implementer（設計と実装計画に基づき src/ へ自律実装。テスト実行含む）
  → code-reviewer（設計適合性・品質・セキュリティのレビュー）
  → 総合評価が「要修正」「差し戻し」→ implementer を自動的に再起動して修正 → 再レビュー
  → 総合評価が「OK」→ 開発完了
```

### 4.1 開発環境の確認（実装着手前の前提条件）
`implementer` を起動する直前に必ず、`.env`・Composer依存関係・Dockerコンテナの
起動状態・DBマイグレーション状態を確認する。1つでも未整備なら
`./scripts/setup.sh` の実行をユーザーに依頼し、ループを止める。

### 4.2 自律ループの停止条件
以下のいずれかに該当した場合のみ、ループを止めてユーザーに報告する。
- レビュー往復が**3回連続**で「OK」に至らない
- 開発環境が未整備
- 設計との齟齬が判明した（`design/questions.md` に記録して確認）
- 未完了タスクが全て依存関係でブロックされている

上記以外（通常のタスク処理・要修正でのやり直し）は確認なしで自動的に進む。

### 4.3 誰が「途中で止まっていないか」を確認するか
ループ実行中は、実行者であるメインチャット自身が上記の停止条件を毎回自己判定する
（独立した監視役が別途いるわけではない）。ループが停止条件に当たらないまま
セッション断絶等で中断した場合、能動的に検知する仕組みは現状ない。ただし
`implementation-plan.md`・`review-report.md`・`feature-log.md` の状態は常に
ファイルに残るため、次にセッションが開始された時点（design/dev 双方の
「セッション開始時の動作」）で必ず正しい状態が検出・報告される
（＝「気づかれない」ことはあっても「壊れた状態のまま消える」ことはない）。

|---|---|---|
| requirements-gatherer | design | 要件定義前の材料収集 |
| requirements-analyst | design | 要件定義の作成 |
| basic-designer | design | 基本設計の作成 |

| db-designer | design | DB設計の作成 |
| diagram-designer | design | 図解（Mermaid）の作成 |

| code-reviewer | dev | 設計適合性・品質・セキュリティのレビュー |

## 6. 完成後の変更経路

一度開発パイプラインが完了した後の変更は、変更の性質によって次の3経路に振り分ける。

### 6.1 設計変更を伴う変更（機能追加・画面追加・テーブル追加・ボタン追加）
規模を問わずすべて design 側の「機能追加フロー」に乗せる（影響範囲判定で
不要な設計ファイルは自然にスキップされるため、規模ごとに別フローは用意しない）。

```
「〇〇機能を追加したい」「〇〇画面が欲しい」「〇〇テーブルを追加したい」「〇〇ボタンが欲しい」
  → メインチャット（機能追加ヒアリング） → design/feature-log.md に記録
  → 影響範囲判定 → 該当する設計エージェントのみ追記モードで起動
  → 全対象「反映済み」→ feature-log.md の状態を「設計反映済み」「開発への反映: 未反映」に更新
  → （ユーザーの追加操作なしで）次回の開発パイプライン起動時に自動検知
  → implementation-planner が追記モードで起動
    （既存の完了チェックは保持したまま、新機能分のタスクだけを追記）
  → 通常の自律ループに合流し implementer が実装、完了したら feature-log.md を「実装完了」に更新
```

### 6.2 設計変更を伴わない直接修正（バグ修正・手動修正）
design/ を経由せず、システム開発パイプライン設計書.md の「アドホック修正
フロー」で直接処理する。

```
「〇〇のバグを直して」「〇〇を修正して」
  → dev/adhoc-fixes.md に記録
  → implementer をアドホック修正モードで起動（最優先で対応）
  → 完了後 code-reviewer で検証
```

修正が実は設計変更を伴うと判明した場合、`implementer` は無断で進めず
ユーザーに報告し、6.1 の機能追加フローに差し戻す。

### 6.3 判断に迷う場合
設計変更を伴うかどうか不明な場合は、`AskUserQuestion` でユーザーに確認してから
6.1・6.2 のどちらに振り分けるか決める。

## 7. テンプレートの同期

`.claude/agents/`・`.claude/skills/` と各 `CLAUDE.md`・`docs/` 配下の各設計書は、
テンプレートリポジトリ `claude-pipeline-template` から配布される「フレーム
ワーク部分」である。同期対象は `.claude/SYNC_MANIFEST` に列挙され、`design/`・
`dev/` の実データや `src/` の実装コード、`docker/`・`docs/`・`scripts/` の
実ファイルは同期対象外（プロジェクト固有のため上書きしない）。

### 7.1 アップストリームへの反映（テンプレート→の逆方向）
上記フレームワーク部分（`.claude/SYNC_MANIFEST` に列挙されたファイル）を
個別プロジェクト側で追加・修正した場合、それはフレームワーク自体の改善で
ある可能性が高い。**メインチャットは、これらのファイルを変更する際、必ず
利用者に「テンプレートリポジトリへ反映するか」を確認してから反映すること**
（無断反映は禁止）。詳細な確認手順は
[claude-pipeline-template/CLAUDE.md](../CLAUDE.md)「テンプレートへの反映」を
参照。開発ソース（同期対象外のファイル）はこの確認フローの対象外であり、
テンプレートへ反映することは一切ない。

## 8. 既知の前提・今後見直す可能性がある点

- レビュー往復の上限「3回」は暫定値。運用してみて多い/少ないと感じたら
  システム開発パイプライン設計書.md の「自律ループの継続条件・停止条件」を
  調整する。カウント自体はファイルに保存されず、メインチャットがそのセッション
  内で自己申告的に数える運用である点にも留意する（`.claude/settings.json` の
  `code-reviewer` フックは毎回同じ案内文を出すだけで、実際の往復回数を
  記録・参照する仕組みは持たない）。
- 自律ループの長時間ハング・セッション断絶を能動的に検知する仕組み
  （Bashコマンドのタイムアウト、中間報告義務等）は未実装。現状は「状態は
  必ずファイルに残る」ことで担保している。
- `docker/` は既に Dockerfile・設定ファイルが整備済みである（詳細は
  `docs/システム開発Docker環境設計書.md`「現在の状態」）。Dockerfile自体を
  生成する専用サブエージェントは存在せず、方針を同書に定義してから人間または
  Claude が直接作成する運用を想定している（今回もその運用で作成済み）。
  `docs/` は空ではなく、本書を含むパイプラインのメタ設計書7点が既に存在する
  （実装済みシステム向けの運用ドキュメントはまだない、という意味では従来の
  記述の意図どおり）。
