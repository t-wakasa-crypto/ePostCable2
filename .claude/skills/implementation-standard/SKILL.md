# 実装計画 品質基準

## フェーズ分け方針

| フェーズ | 内容 |
|---|---|
| Phase 1: 基盤 | 開発環境・DB・認証など、他タスクが依存するもの |
| Phase 2: MVP | 最小限の機能で動作確認できる状態 |
| Phase 3: 基本機能 | requirements.md の必須機能をすべて実装 |
| Phase 4: 拡張機能 | あると良い機能・パフォーマンス改善 |

## タスク記述の形式

```markdown
### [T001] タスク名
- **フェーズ**: Phase 1
- **工数**: 小（〜半日）/ 中（1〜2日）/ 大（3日以上）
- **依存**: T000（前提となるタスクのID）
- **内容**: 何を実装するか（1〜3行）
- **完了条件**: どうなれば完了か（テスト・動作確認の基準）
```

## 優先順位の判断基準
- 他タスクが依存しているものを先にする
- DB・APIサーバー → フロントエンドの順
- 認証・認可は早期に実装する（後から入れると影響範囲が大きい）
- 外部サービス連携は疎結合にしてモック実装から始める

## 品質チェックリスト
- [ ] すべての機能要件がいずれかのタスクに対応しているか
- [ ] タスク間の依存関係に循環がないか
- [ ] Phase 2（MVP）で最低限の動作確認ができる構成か
- [ ] 工数「大」のタスクはさらに分割できないか
